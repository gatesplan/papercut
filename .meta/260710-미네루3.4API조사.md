# MinerU 3.4 API 조사 (2026-07-10)

## 확인된 사항

- 최신 버전 3.4.3 (2026-06-18 배포), Python 3.10~3.13 [근거, https://pypi.org/pypi/mineru/json]
- 백엔드 종류: pipeline, vlm-engine, hybrid-engine, vlm-http-client, hybrid-http-client. 데모 기본값은 hybrid-engine [근거, demo/demo.py]
- Python에서 쓰는 두 가지 방식:
  1. in-process: `mineru.cli.common.do_parse(output_dir, pdf_file_names, pdf_bytes_list, p_lang_list, backend="pipeline", f_dump_* 플래그...)`. 출력은 `{output_dir}/{이름}/{parse_method}/`에 md + images/ [근거, mineru-3.4.3-released 태그 소스]
  2. 로컬 API 서버: `mineru.cli.api_client.LocalAPIServer`가 `python -m mineru.cli.fast_api` 서브프로세스를 띄움. submit_parse_task_sync로 제출, wait_for_task_result(async)로 대기, download_result_zip(async)으로 zip 수신, safe_extract_zip으로 해제 [근거, mineru-3.4.3-released mineru/cli/api_client.py]
- `read_fn(path)`: 파일을 bytes로 읽음, 이미지는 PDF로 변환
- 모델 소스는 env MINERU_MODEL_SOURCE(huggingface/modelscope)로 지정

## 판단

- Extractor는 로컬 API 서버(서브프로세스) 방식 채택.
  - 이유: unload = 서브프로세스 종료 = VRAM/RAM 반납이 OS 수준에서 보장됨. in-process do_parse는 mineru 내부 전역 모델 캐시 때문에 파이썬에서 확실한 해제가 불가능.
- 영향: 가중치 다운로드가 생성자 시점이 아니라 서버측 첫 파싱 시점에 발생. 설계의 "생성자가 가중치 다운로드" 항목 수정 필요.

## 미확인 사항

- [추적필요] 결과 zip의 내부 디렉토리 구조. 구현은 재귀 탐색으로 md/images를 찾는 방식으로 방어.
- [추적필요] 3.x에서 hybrid-engine의 VRAM 요구량 (문서상 pipeline 4GB / vlm 8GB는 확인됨).
